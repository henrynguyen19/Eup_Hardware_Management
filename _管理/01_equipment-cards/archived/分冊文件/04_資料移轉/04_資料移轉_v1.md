## 第 2 章 檔名解析規則

**主照片**：`{equipment_id}_{name}.{ext}`
```
1000057_FUHO AHD 4G-DVR(4路)雙SD.jpg
→ equipment_id = 1000057
→ name = FUHO AHD 4G-DVR(4路)雙SD
```

**細節照片**：`{equipment_id}_{name}_{suffix}.{ext}`

suffix 種類範例：`2`、`3`、`整組`、`舊照`、`配線`、`配線_2`、`花蓮復康_改用這外觀`

```
1000057_FUHO AHD 4G-DVR(4路)雙SD_配線_2.jpg
→ equipment_id = 1000057
→ name = FUHO AHD 4G-DVR(4路)雙SD（從主照片 map 取得）
→ suffix = 配線_2
```

> **解析策略**：先從主照片建立 `equipment_id → name` 對照表，
> 再用「name 前綴比對」剝出細節照片的 suffix。
> 當 name 含底線（如 `4G-DVR`）時此策略才能正確解析。

## 第 4 章 腳本特性

**可重複執行（冪等）**：
腳本啟動時查詢 Supabase，已有 `main_photo` 的料卡自動跳過。
中斷後重跑不重複上傳。

**Supabase 寫入時機**：
主照片 + 所有細節照片全部上傳成功後，才做一次 upsert。
任何一張失敗，整筆記錄不寫入（下次重跑再試）。

**速率限制**：每張上傳間隔 300ms，避免 Cloudinary API rate limit。

## 第 5 章 已知問題與修正

### Windows PNG 上傳失敗

**問題**：Cloudinary SDK v2.10.0 在 Windows 上用**路徑字串**直接上傳 `.png` 檔案時，
回傳 `Invalid URL for upload (HTTP 400)`。同樣路徑的 `.jpg` 不受影響。

**根因**：SDK 在 Windows 路徑處理上的 bug，PNG 與 JPG 走不同分支。

**修正**：改用 `fs.createReadStream` + `upload_stream`，所有格式均穩定上傳。

```typescript
// 修正後的 uploadToCloudinary
function uploadToCloudinary(filePath: string, public_id: string): Promise<UploadApiResponse> {
  return new Promise((resolve, reject) => {
    const writeStream = cloudinary.uploader.upload_stream(
      { folder: FOLDER, public_id, overwrite: false, resource_type: 'image' },
      (error, result) => error ? reject(error) : resolve(result!)
    )
    fs.createReadStream(filePath).pipe(writeStream)
  })
}
```

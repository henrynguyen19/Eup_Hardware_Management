# 第 02 冊：資料分析與 Schema 設計

---

## 第 1 章 現有照片資料夾長什麼樣？

### 1-1 資料夾結構

本地端的設備照片分成兩個資料夾：

```
設備料卡/
├── 設備線材_照片/          ← 主照片（每台設備一張）
│   ├── 1000003_S168-3G衛星定位器.jpg
│   ├── 1000005_S168-4G衛星定位器.jpg
│   └── ...（共 791 張）
│
└── 設備線材_照片_細節/     ← 子照片（每台可有多張）
    ├── 1000005_S168-4G衛星定位器_2.jpg
    ├── 1000041_EP-168 4G環保車機_整組.jpg
    ├── 1000057_FUHO AHD 4G-DVR(4路)雙SD_配線.jpg
    └── ...（共 1095 張）
```

| 資料夾 | 張數 | 說明 |
|--------|------|------|
| 設備線材_照片 | 791 張 | 主照片，一台設備一張代表圖 |
| 設備線材_照片_細節 | 1095 張 | 子照片，包含細節、整組、配線等 |

### 1-2 照片命名規則

**主照片命名格式**：
```
{設備ID}_{設備名稱}.jpg
```
範例：`1000003_S168-3G衛星定位器.jpg`

**子照片命名格式**：
```
{設備ID}_{設備名稱}_{類型或序號}.jpg
```

子照片後綴類型說明：

| 後綴 | 意義 | 範例 |
|------|------|------|
| `_2`、`_3`、`_4` | 第 2、3、4 張照片 | `1000041_EP-168_2.jpg` |
| `_整組` | 整套設備合照 | `1000057_FUHO AHD_整組.jpg` |
| `_配線` | 配線照片 | `1000057_FUHO AHD_配線.jpg` |
| `_配線_2` | 第 2 張配線照片 | `1000057_FUHO AHD_配線_2.jpg` |

> 💡 設備 ID 是 7 位數字，從 `1000003` 開始，這就是我們資料庫的主鍵（唯一識別碼）。

## 第 2 章 資料庫設計的思考邏輯

在設計資料庫之前，先想清楚「每個欄位要解決什麼問題」。

### 2-1 為什麼分類（category）要獨立，不放進 Tags？

因為「主機、天線、支架、螢幕」這類分類是**互斥的**，每台設備只屬於一個主類別。把它獨立成 `category` 欄位的好處：

- 篩選和排序效能更好
- UI 可以做成下拉選單，避免打錯字
- 統計報表方便（例如：主機共幾台）

**Tags** 則保留給彈性標籤，例如同一台設備可同時有「4G」、「含RFID」、「停產」等多個標籤，這些才適合用陣列存。

### 2-2 為什麼備註（notes）不放進 Tags？

Tags 是一個一個的「標籤關鍵字」，而備註是一段說明文字。性質完全不同，混在一起的話：
- 搜尋會變複雜
- UI 不知道怎麼顯示
- 未來維護困難

所以備註獨立成 `text` 欄位，可以自由輸入任何文字。

### 2-3 廠商為什麼先用 Text，不做成獨立的廠商表？

因為現階段主要需求是「查詢設備時知道是哪家廠商」，用文字欄位就夠了。

未來如果需要記錄廠商的聯絡電話、業務姓名等資訊，再把廠商拆成獨立的 `vendors` 表，資料庫結構支援這樣的擴充。

## 第 3 章 資料庫 Schema（表格結構）

### 3-1 最終確認的欄位

資料表名稱：`equipment_cards`

| 欄位名稱 | 資料型別 | 說明 | 範例 |
|----------|----------|------|------|
| `equipment_id` | TEXT（主鍵） | 設備唯一編號 | `1000003` |
| `name` | TEXT | 設備名稱 | `S168-3G衛星定位器` |
| `category` | TEXT | 主分類 | `主機`、`天線`、`支架`、`螢幕` |
| `vendor` | TEXT | 廠商名稱 | `XX科技` |
| `status` | TEXT | 使用狀態 | `active`（在線）或 `discontinued`（停用） |
| `tags` | TEXT[] | 彈性標籤陣列 | `["4G", "含RFID"]` |
| `notes` | TEXT | 用途說明、備註 | `用於貨車定位，需搭配天線` |
| `main_photo` | TEXT | 主照片的網址 | `https://r2.xxx.com/1000003_main.jpg` |
| `detail_photos` | JSONB | 子照片網址陣列 | `["https://..._2.jpg", "https://..._整組.jpg"]` |
| `created_at` | TIMESTAMPTZ | 建立時間（自動填入） | `2026-04-29 10:00:00+08` |
| `updated_at` | TIMESTAMPTZ | 最後更新時間 | `2026-04-29 10:00:00+08` |

### 3-2 欄位補充說明

**`status` 狀態欄位**
目前只有兩個值：
- `active`：設備仍在使用中
- `discontinued`：設備已停用

未來如需新增「維修中」、「開發中」等狀態，直接在 Supabase 更新資料即可，不需要改程式結構。

**`main_photo` 和 `detail_photos`**
照片本身不存在資料庫裡（資料庫存照片很慢又貴）。我們把照片上傳到 Cloudflare R2，然後把「照片的網址」存在這兩個欄位。就像你在 LINE 分享圖片，其實傳的是連結，不是圖片本身。

**`detail_photos` 為什麼用 JSONB？**
因為每台設備的子照片數量不同，有的 1 張、有的 8 張。JSONB 可以彈性存放陣列，不需要事先決定「最多幾張」。
